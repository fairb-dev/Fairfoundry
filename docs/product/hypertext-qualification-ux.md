# Hypertext Qualification UX — Product Design

## Core Concept

The qualification process is an interactive hypertext document where production logs and test specifications are linked together visually. Users upload their existing data in whatever format they have, and the platform creates an annotated, linked view where every measurement is connected to its acceptance criteria.

No schema knowledge required. No field mapping forms. No configuration wizards. Just: upload your data, highlight what matters, define what "good" looks like.

---

## The Three Inputs

### 1. Production Logs
Factory uploads raw test data files in any common format:
- `.csv` — most common, exported from ATE, MES, or Excel
- `.json` — from modern test stations or API-connected equipment
- `.xml` — from legacy systems, MTConnect, OPC UA

These are the actual measurement records from the production line. Each row is a unit tested. Each column is a parameter measured.

### 2. Test Specifications
OEM uploads the acceptance criteria in whatever form they exist:
- PDF or Word document (the traditional ERS)
- Image or scan of a spec sheet
- Plain text typed directly
- Structured data (Excel with limits, JSON with criteria)

These define what "good" looks like. The AI agent extracts structured criteria from unstructured documents.

### 3. The Hypertext (generated by the platform)
The platform creates an interactive linked view where:
- Production log data is rendered as an annotated table
- Columns are highlighted and categorized (measurement, condition, identifier, equipment)
- Each measurement column is linked to its acceptance criteria
- Each criteria is linked to its source in the ERS document
- Pass/fail is calculated and displayed inline
- Both OEM and factory see the same view

## Step-by-Step Flow

### Step 1: Upload Production Log

Factory drops a file. The platform renders it immediately as a table.

```
┌──────────┬────────────────────┬────────────┬────────────┬────────┬────────┬────────┬─────────┬──────────┐
│ serial   │ timestamp          │ mtf_center │ mtf_corner │ sfr_h  │ sfr_v  │ temp_c │ station │ operator │
├──────────┼────────────────────┼────────────┼────────────┼────────┼────────┼────────┼─────────┼──────────┤
│ SN001    │ 2026-04-11T08:23   │ 0.42       │ 0.31       │ 0.38   │ 0.36   │ 24.2   │ STN-04  │ J.Kim    │
│ SN002    │ 2026-04-11T08:24   │ 0.39       │ 0.28       │ 0.35   │ 0.33   │ 24.3   │ STN-04  │ J.Kim    │
│ SN003    │ 2026-04-11T08:25   │ 0.44       │ 0.33       │ 0.40   │ 0.38   │ 24.1   │ STN-04  │ J.Kim    │
│ SN004    │ 2026-04-11T08:26   │ 0.36       │ 0.22       │ 0.32   │ 0.30   │ 24.4   │ STN-04  │ J.Kim    │
└──────────┴────────────────────┴────────────┴────────────┴────────┴────────┴────────┴─────────┴──────────┘
```

### Step 2: AI Agent Suggests Column Roles

The AI agent scans column headers, data types, and value distributions to suggest what each column represents:

| Column | AI Suggestion | Confidence | Category |
|--------|--------------|------------|----------|
| `serial` | Unit identifier | 95% | Identity |
| `timestamp` | Test timestamp | 99% | Metadata |
| `mtf_center` | MTF measurement (center field) | 88% | Measurement |
| `mtf_corner` | MTF measurement (corner field) | 87% | Measurement |
| `sfr_h` | SFR measurement (horizontal) | 82% | Measurement |
| `sfr_v` | SFR measurement (vertical) | 82% | Measurement |
| `temp_c` | Temperature (°C) | 91% | Test condition |
| `station` | Equipment identifier | 85% | Equipment |
| `operator` | Operator/tester | 78% | Metadata |

The user reviews and confirms or corrects. One click per column. If the AI is wrong, the user picks from a dropdown: Measurement, Test Condition, Identity, Equipment, Metadata, Ignore.

### Step 3: Upload Test Specification (Optional — can also define manually)

OEM uploads their ERS document. AI agent extracts structured criteria:

```
Extracted from "Camera_Module_ERS_v2.3.pdf":

Section 3.2.1: MTF at center field position ≥ 0.35 cy/px
  Condition: 25°C ± 2°C ambient temperature
  
Section 3.2.2: MTF at corner field position (0.7 field height) ≥ 0.25 cy/px
  Condition: same as 3.2.1
  
Section 3.3.1: SFR horizontal ≥ 0.30 lp/px
  Reference: ISO 12233 method
  
Section 3.3.2: SFR vertical ≥ 0.30 lp/px
  Reference: ISO 12233 method
  
Section 4.1: Test temperature: 25°C ± 2°C
  Must be recorded per unit
```

Each extracted criteria appears as a linkable card in the interface.

### Step 4: Link Columns to Criteria (the Hypertext Moment)

Two ways to create links:

**Option A — AI auto-match:** The AI suggests matches based on name similarity, value ranges, and context:
- `mtf_center` ↔ Section 3.2.1 (MTF center ≥ 0.35) → 92% confidence
- `mtf_corner` ↔ Section 3.2.2 (MTF corner ≥ 0.25) → 90% confidence
- `sfr_h` ↔ Section 3.3.1 (SFR horizontal ≥ 0.30) → 85% confidence

User confirms with one click per match.

**Option B — Manual drag-and-drop:** User drags a criteria card onto a column header. A panel opens:

```
┌─────────────────────────────────────────────┐
│ Linking: mtf_center ↔ ERS Section 3.2.1     │
│                                             │
│ Parameter name:  MTF at center field        │
│ Criteria type:   ≥ (greater than or equal)  │
│ Limit:           0.35                       │
│ Unit:            cy/px                      │
│ Condition:       temp_c within 23-27°C      │
│ Source:          Section 3.2.1, page 4      │
│                                             │
│ [Confirm Link]                              │
└─────────────────────────────────────────────┘
```

**Option C — Manual without ERS:** User clicks a column header and types the criteria directly:
- Lower limit: 0.35
- Unit: cy/px
- Note: "MTF at center per customer spec"

No ERS document needed. The criteria is defined inline.

### Step 5: The Hypertext View

Once columns are linked to criteria, the table transforms:

```
┌──────────┬────────────────────┬──────────────┬──────────────┬──────────┬──────────┬────────┐
│ serial   │ timestamp          │ mtf_center   │ mtf_corner   │ sfr_h    │ sfr_v    │ temp_c │
│          │                    │ ≥0.35 cy/px  │ ≥0.25 cy/px  │ ≥0.30    │ ≥0.30    │ 23-27  │
│          │                    │ §3.2.1       │ §3.2.2       │ §3.3.1   │ §3.3.2   │ §4.1   │
├──────────┼────────────────────┼──────────────┼──────────────┼──────────┼──────────┼────────┤
│ SN001    │ 2026-04-11T08:23   │ ✓ 0.42       │ ✓ 0.31       │ ✓ 0.38   │ ✓ 0.36   │ ✓ 24.2 │
│ SN002    │ 2026-04-11T08:24   │ ✓ 0.39       │ ✓ 0.28       │ ✓ 0.35   │ ✓ 0.33   │ ✓ 24.3 │
│ SN003    │ 2026-04-11T08:25   │ ✓ 0.44       │ ✓ 0.33       │ ✓ 0.40   │ ✓ 0.38   │ ✓ 24.1 │
│ SN004    │ 2026-04-11T08:26   │ ✓ 0.36       │ ✗ 0.22       │ ✓ 0.32   │ ✓ 0.30   │ ✓ 24.4 │
└──────────┴────────────────────┴──────────────┴──────────────┴──────────┴──────────┴────────┘

Summary: 3/4 units pass all criteria. SN004 fails mtf_corner (0.22 < 0.25, ref §3.2.2)
```

Key features of the hypertext view:
- Column headers show the acceptance criteria and ERS source section
- Each cell shows ✓ (pass) or ✗ (fail) next to the value
- Clicking any ✗ cell shows: the measured value, the limit, the ERS source, and the gap
- Clicking a section reference (§3.2.1) opens the relevant section in the ERS document
- The summary bar shows overall pass rate

### Step 6: Both Sides Review in Sandbox

The hypertext view is shared between OEM and factory. Both sides see:
- The same data
- The same criteria
- The same pass/fail results
- The same ERS source references

Either side can:
- Propose changes to criteria (requires mutual agreement)
- Add conditions (e.g., "temperature must be within range for result to count")
- Add new measurement columns
- Mark columns as "for reference only" (recorded but not used for pass/fail)

When both sides agree that the hypertext accurately represents their quality agreement, they lock it and transition to production.

### Step 7: Production Mode

In production, the same flow runs automatically:
1. Factory uploads production log (same format as sandbox)
2. Platform applies the locked hypertext mapping
3. Pass/fail calculated per the agreed criteria
4. Results visible to both sides
5. 72-hour review window
6. Settlement triggers automatically

If the factory changes their CSV format (adds a column, renames a header), the AI agent flags the change and suggests an updated mapping. Both sides approve before it takes effect.

## Non-Numeric Criteria in Hypertext

Not all acceptance criteria are "number above/below a limit." The hypertext view supports:

### Image-based criteria
- Factory uploads images (e.g., cosmetic inspection photos)
- Criteria: "No visible scratches" or "AI classification confidence ≥ 0.9"
- Hypertext shows thumbnails with ✓/✗ overlay
- Clicking opens full image with defect annotations (if any)

### Categorical criteria
- Column contains text values: "PASS", "FAIL", "REWORK"
- Criteria: accepted values = ["PASS"]
- Hypertext shows green/red based on string match

### Pattern/waveform criteria
- Factory uploads waveform data (frequency response, impedance curves)
- Criteria: "correlation with golden reference ≥ 0.95" or "within mask limits"
- Hypertext shows mini waveform plot with ✓/✗
- Clicking opens full waveform comparison view

### Composite criteria
- Multiple columns combine into a single pass/fail
- Example: "Overall score = 0.4 × mtf_center + 0.3 × sfr_h + 0.3 × sfr_v, must be ≥ 0.35"
- Hypertext shows individual values plus computed composite with ✓/✗

## AI Agent Roles in the Hypertext Flow

### Spec Parser Agent
- Input: ERS document (PDF, Word, image, text)
- Output: Structured criteria cards (parameter, limit, unit, condition, source reference)
- Accuracy improves with more ERS documents seen across the platform
- User always reviews and confirms

### Log Mapper Agent
- Input: Production log file (CSV, JSON, XML)
- Output: Column role suggestions (measurement, condition, identity, equipment, metadata)
- Uses column names, data types, value distributions, and patterns from other factories
- User always reviews and confirms

### Match Agent
- Input: Mapped columns + extracted criteria cards
- Output: Suggested links between columns and criteria
- Uses semantic similarity, unit matching, value range overlap
- User always reviews and confirms

### Drift Agent (production mode)
- Monitors incoming data for:
  - Format changes (new columns, renamed headers)
  - Distribution shifts (values trending toward limits)
  - Condition violations (temperature out of spec range)
- Flags issues before they cause failures
- Does not make autonomous decisions — alerts both parties

## What the User Never Sees

Behind the hypertext, the platform generates:
- Universal schema records (JSON) for every unit tested
- Qualification criteria encoded in the settlement contract
- Merkle root commitments for tamper-proof data integrity
- Settlement calculations (escrow, fees, payment amounts)
- Quality credential metadata for verified completions

All of this is invisible. The user interacts with a highlighted spreadsheet and a document viewer. The blockchain, the schema, the settlement logic — all hidden.

---

## Design Principles

1. **The production log IS the interface.** Don't make users learn a new tool. Show them their own data.
2. **AI suggests, humans confirm.** Every AI suggestion requires one click to accept. No autonomous decisions.
3. **The hypertext IS the contract.** What both sides see is what gets enforced. No separate "terms and conditions" document.
4. **Progressive disclosure.** Simple view by default. Click to see details. Click again to see the source. Never force complexity on the user.
5. **Any format in, universal format out.** The platform adapts to the factory's data, not the other way around.
